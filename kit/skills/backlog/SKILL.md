# /backlog — Cross-sub-project backlog compiler

Aggregate backlog state from every registered sub-repo into a single
sorted view. Writes `state/backlog-compiled.md`, surfaces a chat
summary.

**Backlog scope only** — queued work, not in-progress. For active
work use `/tasks`. For phase-level forward planning use `/roadmap`.
The split:

| Skill | Scope |
|---|---|
| `/roadmap` | Future phases — strategic planning |
| `/backlog` | Items queued, not yet started — tactical queue |
| `/tasks` | Active work (in-progress + next up) |

## Output pattern

[Pattern 4 — Sprint task board](../../output-catalogue.md#4--sprint-task-board)
adapted for cross-repo: per-sub-repo group with item count, priority
mini-progress dots, expandable item list. The compiled markdown
artifact uses Pattern 4 framing; the chat summary uses Pattern 28
(Stats card grid) for headline counts.

## Behavior contract

- **Read-only.** Reads from `repos/<name>/`; no writes into
  sub-repos.
- **Surface stale data.** Same freshness gate as `/roadmap` and
  `/tasks` — warn if `state/last-fetch.json` is missing or >24h.
- **Skip remote-only sub-repos.** If `repos/<name>/` doesn't exist,
  mark `unfetched`, skip rendering.
- **Honest about source variation.** Kit-enabled repos have
  rich backlog data with frontmatter; non-kit repos may have just a
  flat `BACKLOG.md` or only GitHub issues. Render what's actually
  there.
- **Stale-item flag.** A backlog item that's been queued for ≥60
  days without movement → flag with ⚠ "aging."

## Sources per sub-repo (priority order)

For each `repos/<name>/`:

1. **claude-kit backlog directory:** `repos/<name>/tasks/backlog/*.md`
   — each file is one backlog item. Parse frontmatter for `id`,
   `title`, `priority` (if present), `created`, `assignee`.
2. **claude-kit ROADMAP "next up" items:** entries in
   `repos/<name>/tasks/ROADMAP.md` under phases that haven't
   started, marked as queued.
3. **Top-level BACKLOG.md:** `repos/<name>/BACKLOG.md` — fallback
   for non-kit repos. Plain markdown list.
4. **GitHub issues with "backlog" label** *(further fallback)*:
   `gh issue list -R <owner>/<repo> --label backlog --state open
   --json number,title,createdAt,labels`. Only used if local
   sources empty AND last-fetch was recent.
5. **Nothing found:** mark as `no backlog source`.

## Process

1. Read `state/manifest.md` for the registered list.
2. For each sub-repo, walk source priority and collect backlog
   items with: id (or filename slug), title, priority (if known),
   created date, source (which of the above types).
3. Compute stale flag: items with `created` ≥ 60 days ago.
4. Sort within each sub-repo:
   - Priority high → medium → low → unspecified
   - Within priority: oldest first (FIFO)
5. Group by sub-repo for the compiled artifact.
6. Surface a cross-stack chat summary.

## Style rules

- **Compact lists.** One line per backlog item. No paragraphs.
- **Priority dots are TL;DR.** Reader scans dots before reading
  titles.
- **Stale flag is load-bearing.** ⚠ aging surfaces what's been
  queued too long — those items either need to move or be killed.
- **Honest about source variation.** Non-kit repos with thin
  backlog data say so explicitly.
- **No emoji decoration** beyond ⚠ for aging.

## What you must NOT do

- **Don't write into sub-repos.** Read-only.
- **Don't auto-fetch.** Warn on stale data; don't trigger
  `/refresh`.
- **Don't fabricate priorities.** If a non-kit source has no
  priority data, render under "unspecified" or "no priority data" —
  don't invent.
- **Don't recite items in active or done.** Backlog scope only —
  skip `tasks/active/` and `tasks/done/` (those are in `/tasks`'s
  scope).
- **Don't promise sequencing.** "Coupling opportunity" notes are
  signals; sequencing is the user's call (or `/feature` /
  `/migration`'s).

## When NOT to use this skill

- **Want active in-progress work** → `/tasks`.
- **Want strategic forward roadmap** → `/roadmap`.
- **Want everything (status snapshot)** → `/status`.
- **Want depth on one repo's backlog** → open
  `repos/<name>/tasks/backlog/` directly.
- **Want to file a new backlog item from the orchestrator** →
  that's a `/migration` or `/feature` task drop, not this skill.

## What "done" looks like

`state/backlog-compiled.md` regenerated. Chat summary surfaces
per-repo counts + cross-stack signals + aging items. No commits,
no remote calls beyond cached data.
