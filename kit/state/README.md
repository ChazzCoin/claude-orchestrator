# State

Files in this directory hold **reflected state** — the orchestrator's
view of what's currently true across registered sub-repos. Mostly
regenerated from authoritative sources (`git`, `gh`, sub-repo
`.claude/active.md`); hand-authored truth lives elsewhere (contracts,
decisions, conventions, etc.).

## Contents

| File | Owner | Description |
|---|---|---|
| `manifest.md` | hand-authored at register time, then mostly stable | Index of registered sub-repos: name, role, path, remote, default branch, kit-enabled flag, last-synced metadata. See [`bootstrap/state/manifest.md.template`](../../bootstrap/state/manifest.md.template) for the format. |
| `sub-repos/<name>.md` | mostly auto-managed by `/sync-check`; **Notes** section is hand-authored and preserved across sync | Per-sub-repo accumulated state. One file per registered sub-repo. Holds: current branch / HEAD, advertisement read from sub-kit, every orchestrator artifact (migration / feature / ADR / risk / incident) that touches this repo, capability flags. See [`sub-repos/_template.md`](sub-repos/_template.md) for the schema. |
| `sync-status.md` | regenerated by `/sync-check` | Cross-sub-repo snapshot: per sub-repo branch / HEAD / open PRs, per active migration per-repo state, drift flags. Don't edit; regenerate. |

## Manifest vs. per-sub-repo file

The discipline:

- **`manifest.md`** is the **index** — short, one entry per sub-repo,
  minimal metadata, plus pointers into `sub-repos/<name>.md` for depth.
  Stable across sync.
- **`sub-repos/<name>.md`** is the **depth** — accumulated context per
  sub-repo, refreshed by `/sync-check`. Has a **Notes** section the user
  hand-edits (preserved across sync).
- **`sync-status.md`** is the **cross-sub-repo snapshot** — point-in-time,
  not durable. Regenerated wholesale.

If you want to know "what's the state of <api> right now", you read
`sub-repos/api.md` (richest). If you want "is anything in drift across
the stack," you read `sync-status.md`. If you want "what sub-repos
even exist," you read `manifest.md`.

## Why this is its own directory

Two reasons:

1. **Don't conflate authored truth with reflected state.** Authored
   files (contracts, decisions, conventions, vision, principles) live
   in the world of "what we decided." Reflected state lives in "what's
   currently true." Mixing them confuses both.
2. **Regeneration is cheap and authoritative; hand-maintenance rots.**
   `gh` and `git log` are source of truth for repo state. The
   orchestrator queries them on demand rather than pretending to track
   them.

## Update flow

```
/register
  → adds entry to manifest.md
  → creates sub-repos/<name>.md from sub-repos/_template.md
  → detects kit-enabled (presence of <sub>/.claude/foundation.json)

/sync-check
  → reads each manifest entry
  → fetches each sub-repo (no pull, no merge)
  → reads <sub>/.claude/active.md if present
  → updates sub-repos/<name>.md (auto fields; preserves Notes section)
  → regenerates sync-status.md wholesale
  → flags drift loudly

/migration, /feature, /decision, /risk, /incident — each, when filing
  → updates the appropriate "Affecting this repo" section in
    sub-repos/<name>.md for each affected sub-repo (additive; never
    deletes manual notes)
```

## Generated vs durable

Files in this directory split:

- **Durable (committed):** `manifest.md`, `sub-repos/<name>.md`
- **Generated (gitignored in instances by default):** `sync-status.md`,
  `audit-progress.md`, `brief-cache.md`

The instance's `.gitignore` (created by `bin/init`) ignores the
generated ones so they don't pollute history.
